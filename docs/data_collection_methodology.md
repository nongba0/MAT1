# 맛집 데이터 수집 및 정제 방법론 (Data Collection & Refinement Methodology)

본 문서는 현재 시스템에 적재된 맛집(라멘 및 일반 한식/퓨전 등) 데이터를 수집, 정제, 그리고 시스템에 반영하는 일련의 파이프라인과 철학을 기록한 문서입니다.

## 1. 데이터 수집 원칙 (Data Persona)
데이터는 **냉철한 미식 데이터 엔지니어**의 시각으로 수집 및 분석됩니다.
- **배제 대상**: "분위기가 좋아요", "사장님이 친절해요", "가성비가 최고예요" 등 감정적, 주관적, 광고성 표현은 철저히 배제합니다.
- **추출 대상**: 요리의 조리법, 육수의 질감, 메인 식재료의 특징, 서빙 방식(예: 토렴식), 객관적인 미식 가이드 선정 여부(예: 미쉐린 가이드) 등 **미식적(Gustatory) 요소**와 **객관적 특징**만을 추출합니다.

## 2. 데이터 분석 및 추출 프로세스
새로운 음식점이 시스템에 추가될 때마다 다음의 과정을 거쳐 데이터화됩니다.

1. **리뷰 데이터 교차 검증**: 한 음식점 당 최소 10개 이상의 블로그 및 방문자 리뷰를 크로스 체크합니다.
2. **핵심 특징 요약 (Description)**: 
   - 리뷰에서 반복적으로 언급되는 객관적 맛의 특징을 1~3문장으로 압축합니다.
   - *예시: "미쉐린에 빛나는 돼지 곰탕의 선구자입니다. 밥이 말아져 나오는 토렴식이며, 국물 점도는 물처럼 가벼운데 감칠맛은 폭발적이라는 찬사가 쏟아집니다."*
3. **대표 메뉴 설정 및 상세 설명 추출 (Signature Menus & Menu Description)**:
   - **메뉴 선정 기준**: 식당의 정체성을 가장 잘 보여주는 1~3가지의 대표 메뉴를 선정합니다. (예: 타코집의 경우 '까르니따스', 국밥집의 경우 '돼지국밥, 수육')
   - **설명 작성 원칙**: 메뉴의 맛을 평가하는 주관적인 단어("환상적인", "너무 맛있는")를 철저히 배제합니다.
   - **기술 요소**: 오직 "사용된 핵심 식재료", "독특한 조리 공법(예: 라드에 저온 조리, 토렴, 10시간 끓임)", "객관적인 텍스처와 형태(예: 두꺼운 면, 맑은 육수, 묵직한 크림)" 만을 1~3문장으로 간결하게 서술합니다.
   - *예시: "메뉴: 대파 크림치즈 바게트, 맘모스빵 / 설명: 버터에 볶은 대파와 갈릭 크림치즈를 하드 계열 바게트에 채웠으며, 맘모스빵은 팥 앙금과 밤다이스를 매우 두껍게 넣은 묵직한 스타일입니다."*
4. **키워드 태그 추출 (Keyword Tags)**:
   - 요약된 특징 중 가장 핵심이 되는 단어 3~5개를 추출하여 해시태그 형태로 저장합니다.
   - *예시: `#돼지곰탕`, `#토렴식`, `#감칠맛폭발`*

## 3. 알고리즘 로직과 뷰(View) 데이터의 완전한 분리
데이터베이스 모델링과 화면 표시 영역을 명확하게 분리하여 유연성을 극대화했습니다.

### A. 매칭 알고리즘용 메타데이터 (Backend Enums)
사용자의 취향 퀴즈 분기점을 처리하기 위해 음식점은 수학적 카테고리로 맵핑됩니다. 이 데이터는 화면에 노출되지 않으며 오직 **분기 매칭 계산**에만 사용됩니다.
- `SoupCategory` (국물 베이스: CLEAR_SOUP, RICH_SOUP, JEON 등)
- `DetailStyle` (상세 형태: KALGUKSU, SUJEBI, GUKBAP, JJIGAE, SHIO, TONKOTSU 등)
- `MainIngredient` (핵심 재료: PORK, SEAFOOD, CHICKEN, POTATO 등)

### B. 사용자 표시용 데이터 (Frontend View)
사용자가 퀴즈 결과를 받았을 때 화면에 렌더링되는 것은 오직 정제된 **고유 설명(Description)**과 **해시태그(Keyword Tags)** 뿐입니다.
이를 통해 알고리즘 로직이 변경되더라도 프론트엔드의 UI는 영향을 받지 않으며, 언제든 음식점별 특징 문구만 독립적으로 수정할 수 있습니다.

## 4. 데이터 자동화 및 주입(Seeding) 파이프라인
수동 타이핑으로 인한 자바 문법(Syntax) 에러를 방지하고, 80여 개가 넘는 식당 데이터를 효율적으로 관리하기 위해 파이썬 기반의 제너레이터(Generator)를 사용합니다.

1. **`generate_seeder.py` (Python)**:
   - 음식점 리스트, 카테고리 매핑, 추출된 Description과 Tags를 파이썬 딕셔너리와 리스트로 관리합니다.
   - 스크립트를 실행하면 Java 문법에 맞는 `DataSeeder.java` 파일을 자동으로 텍스트 렌더링하여 덮어씁니다.
2. **`DataSeeder.java` (Spring Boot)**:
   - Spring Boot의 `CommandLineRunner`를 구현하여, 서버 구동 시 파이썬이 생성해 준 코드를 바탕으로 JPA를 통해 PostgreSQL DB에 데이터를 일괄 `INSERT` 혹은 `UPDATE` 합니다.
3. **Schema 관리 및 UI 동적 렌더링**:
   - `Restaurant.java` 엔티티에 `signatureMenus`, `menuDescription` 등의 신규 필드를 확장할 경우, Spring Data JPA의 `ddl-auto=update` 속성을 통해 데이터베이스 테이블 스키마를 유연하게 동기화합니다.
   - 확장된 메뉴 데이터는 프론트엔드 UI의 결과 카드에서 툴팁 아이콘(❔) 형태로 조건부 렌더링되며, 데이터가 완벽하게 분리되어 있어 언제든 내용을 변경해도 UI 레이아웃이 깨지지 않는 견고한 구조를 가집니다.

---
**[결론]**
이러한 방식을 통해, 현재 시스템은 **감정이 배제된 순도 높은 미식 데이터**를 **결합도가 낮은 구조**로 관리하며, 언제든 새로운 식당 100개, 1000개를 추가하더라도 파이썬 제너레이터를 통해 **무중단에 가까운 속도로 DB를 갱신**할 수 있는 파워풀한 데이터 수집 파이프라인을 구축하고 있습니다.
